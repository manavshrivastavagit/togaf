## Quick Reference Mapping Table

| Page | Core Topic | Learning Objective / Exam Focus | Slide Description |
| :--- | :--- | :--- | :--- |
| **Page 17** | **Enterprise Definition** | Describe what an enterprise is | Defines an enterprise as a collection of organizations with common goals, noting it can be multiple organizations, include partners/suppliers, and maintain multiple EAs. |
| **Page 19** | **Architecture Domains** | List the four subsets of EA (BDAT) | Introduces Business, Data, Application, and Technology (BDAT) architecture domains, explaining their individual scopes and combos (e.g. Security). |
| **Page 25** | **Scoping Architectures** | Explain the 4 dimensions of scoping | Details Breadth (Enterprise Scope), Depth (Level of Detail), Architecture Domains, and Planning Horizon (Time Period), and constraints like people and finance. |
| **Page 26** | **Architecture Levels** | List the 3 levels of Architecture Landscape | Outlines Strategic Architecture (executive), Segment Architecture (program/portfolio), and Capability Architecture (capability increments). |
| **Page 30** | **Abstraction Levels** | Describe how abstraction is used in EA | Explains layering to cross domains using Contextual, Conceptual, Logical, and Physical abstractions (from high-level models to details). |
| **Page 32** | **Building Block - Def** | Explain what building blocks are | Defines Building Blocks (reusable components delivering architectures/solutions), types in metamodel, and their impact on integration/flexibility. |
| **Page 33** | **Building Block - Char** | Describe the characteristics of a building block | Explains criteria for a good building block (reusable, replaceable, specified boundaries, loosely coupled, stable interfaces, subassembly capability). |
| **Page 35** | **TOGAF Introduction** | Explain suitability of the standard | Introduces TOGAF as a best-practice framework for EA, suitable for any context, agile enterprise, digital transformation, and standard cycle of change. |
| **Page 36** | **TOGAF Suitability** | Explain why TOGAF is suitable as a framework | Details benefits (efficient operation, workable solutions, de-risking development, consistency, and stakeholder alignment). |
| **Page 37** | **Tailoring & Integration** | Describe how to tailor TOGAF | Discusses how the generic framework is customized, integrated with other frameworks (ITIL, COBIT, PRINCE), or used standalone. |
| **Page 38** | **Core Structure** | Describe where guidance on applying is provided | Maps the Fundamental Content of the TOGAF Standard to the 6 core documents (Introduction, ADM, Techniques, Guidelines, Content, Capability/Governance). |
| **Page 40** | **The TOGAF Library** | Describe additional guidance portfolio | Explains the TOGAF Library, its templates, patterns, and reference materials, maintained by the Architecture Forum. |
| **Page 41** | **EA Purpose (1/2)** | Explain the purpose of EA | Outlines how EA manages complexity/risk, optimizes processes, provides strategic context for digital capability, and links strategy to value. |
| **Page 42** | **EA Purpose (2/2)** | Explain how EA governs change | Details current/future state mapping, gap analysis, process transparency for personal data, trade-off resolution, and governance. |
| **Page 43** | **EA Benefits** | List the business benefits of EA | Summarizes 6 key benefits: strategic decision-making, efficient operations, digital transformation, ROI/reduced risk, cheaper procurement, and balance. |
| **Page 45** | **ADM Overview (1/9)** | Describe ADM lifecycle | Introduces ADM as the core of the TOGAF standard: a tested, repeatable, iterative process for developing and realizing architecture. |
| **Page 46** | **ADM Overview (2/9)** | Describe ADM structure and adaptation | Details the 10 phases (8 in a cycle), divided into steps, non-waterfall nature, and reference model characteristics. |
| **Page 47** | **Preliminary Phase** | Describe Preliminary Phase purpose | Introduces preparation/initiation to create Architecture Capability, customize framework, and define principles. |
| **Page 48** | **Phase A: Vision** | Describe Phase A purpose | Initial phase of cycle: defines scope, identifies stakeholders, creates vision, and obtains approval to proceed. |
| **Page 49** | **Phases B, C, & D** | Describe Domain Phases | Focuses on developing baseline and target architectures for Business (Phase B), Information Systems (Phase C: Data/Application), and Technology (Phase D). |
| **Page 50** | **Phases E & F** | Describe Planning Phases | Phase E (Opportunities & Solutions) handles initial implementation planning/delivery vehicles; Phase F (Migration Planning) creates migration plan. |
| **Page 51** | **Phases G & H** | Describe Governance/Change Phases | Phase G (Implementation Governance) provides oversight; Phase H (Architecture Change Management) manages change post-deployment. |
| **Page 52** | **Requirements Mgmt** | Describe Requirements Management | Operates continuously throughout the ADM, handling changes through appropriate governance. |
| **Page 53** | **ADM Outputs** | Describe output generation/status | Explains how each phase generates outputs, how their statuses are tracked, and how early outputs are modified in later phases. |
| **Page 56** | **Deliverables vs. Artifacts** | Explain difference of draft/approved deliverables | Explains version numbering, deliverables vs. artifacts, draft status (no review), and approved status (reviewed/approved but still subject to change control). |
| **Page 57** | **Stakeholder** | Define the stakeholder concept | Defines a stakeholder as an individual, team, organization, or class having an interest in a system. |
| **Page 58** | **Concern** | Define the concern concept | Defines a concern as an interest in a system relevant to stakeholders (performance, reliability, security, availability). |
| **Page 59** | **Stakeholder & Concern** | Explain stakeholder/concern relationship | Details how identifying concerns helps ensure stakeholder interests are met and maps concerns to requirements. |
| **Page 60** | **Architecture Viewpoint** | Define architecture viewpoints | Defines viewpoints as vantage points/perspectives defining conventions (schemas) to address specific concerns. Reusable in libraries. |
| **Page 61** | **Architecture View** | Define architecture views | Defines a view as a representation of a system addressing a related set of concerns, consisting of models, specific to a system. |
| **Page 62** | **Viewpoint & View Rel** | Explain view/viewpoint relationship | Explains that every view is rooted in/described by an associated viewpoint. |
| **Page 64** | **Preliminary Phase Purpose** | Explain purpose of Preliminary Phase | Creates Architecture Capability, tailors TOGAF, and defines Architecture Principles. |
| **Page 65** | **Preliminary Phase Obj** | Describe objectives of Preliminary Phase | 1. Determine desired Architecture Capability (context, scope, frameworks, maturity). 2. Establish capability (organizational model, governance, tools, principles). |
| **Page 68** | **Principles Definition** | Describe Architecture Principles | Defines principles as general rules/guidelines (Business, Data, Application, Technology) forming the basis for decisions. Enduring and seldom amended. |
| **Page 69** | **Principles Development** | Explain how principles are developed | Discusses development with stakeholders, approval by Architecture Board, amendment process, and championing by senior management. |
| **Page 72** | **Principles Template** | Explain recommended principle template | Standard 5-element structure: Name, Statement, Rationale (benefits/trade-offs), and Implications (resources, impact, cost). |
| **Page 74** | **Principles Characteristics** | Explain criteria for a good set of principles | Defines 5 criteria: Complete, Consistent, Stable, Understandable, and Robust. |
| **Page 76** | **Principles ADM Usage** | Describe principles usage in ADM phases | Map: Created in Preliminary; reviewed/confirmed in Phase A; aligned/validated in B, C, D. |
| **Page 79** | **Phase A Purpose** | Explain the purpose of Phase A | Starts cycle, defines scope (problem/project), identifies stakeholders, creates Vision, and gets approval to proceed. |
| **Page 80** | **Phase A Objectives** | Describe the objectives of Phase A | 1. Develop high-level aspirational vision. 2. Obtain approval for Statement of Architecture Work. |
| **Page 84** | **Request for Arch Work** | Describe Request for Architecture Work | Defines this official high-level document triggering an ADM cycle. Lists purpose and content (goals, sponsors, IT description). |
| **Page 86** | **Request ADM Usage** | Describe Request usage in ADM | Created in Preliminary (sponsoring org); input for Phase A, B, C, D, E, F, G, H. Generated in Phase F/H under change request. |
| **Page 88** | **Communications Plan** | Explain Communications Plan | Defines how information reaches stakeholders. Includes stakeholders, messages, mechanisms (meetings, newsletters), and timetable. |
| **Page 90** | **Comm Plan ADM Usage** | Describe Comm Plan usage in ADM | Created in Phase A; used as input throughout Phases B, C, D, E, F. |
| **Page 92** | **Business Scenarios** | Explain Business Scenarios | A technique to derive EA characteristics from high-level business requirements to achieve desired outcomes. |
| **Page 93** | **Scenarios Key Elements** | List key elements of Business Scenario | Lists problem, environment, outcomes, human actors, and computer actors. Incomplete examination risk. |
| **Page 94** | **SMART Outcomes** | Explain SMART criteria for scenarios | Specific, Measurable, Actionable, Realistic, and Time-bound outcomes. |
| **Page 95** | **Scenarios Development** | Describe scenario development phases | Outlines 3 successive phases (capture, examine, improve) repeated iteratively. |
| **Page 96** | **Scenarios Diff** | Differentiate Business Scenarios | Compares scenarios with Use-cases (software), Business Models (strategic), SPIN (sales), and Engineering specs. |
| **Page 100** | **Requirements Spec** | Describe Architecture Requirements Spec | Quantitative companion to qualitative Architecture Definition Document. Outlines what implementation projects must do to conform. |
| **Page 102** | **Req Spec ADM Usage (1)** | Describe Req Spec usage in domains | Drafted in Phase A, B, C, D (compiled domain-specific requirements). |
| **Page 103** | **Req Spec ADM Usage (2)** | Describe Req Spec usage in governance | Finalized/used in Phase G (oversight), H (change outcomes), and updated in Requirements Management. |
| **Page 105** | **Business Context** | Describe Business Principles, Goals, Drivers | Provides business context (principles, goals, drivers) outside of architecture discipline but driving it. |
| **Page 107** | **Business Context ADM** | Describe Business Context usage in ADM | Confirmed in Phase A; used as input to select/validate resources in Phases B, C, D. |
